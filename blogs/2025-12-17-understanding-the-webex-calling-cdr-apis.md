<img alt="Understanding the Webex Calling CDR APIs" src="https://images.contentstack.io/v3/assets/bltd74e2c7e18c68b20/bltb0dcf8790b16de3d/6942cabb7d8e3584fa711149/cdr_blog_banner.png?width=900&amp;height=317&amp;fit=crop" /> <p>Until now, there has been only one API that allowed developers to access the raw Call Detail Records (CDRs) from the Webex Calling platform. That API is known, according to the documentation, as the “<strong>Detailed Call History</strong>” API and was accessed with the <a>/v1/cdr_feed</a> endpoint.</p>
<p>This month, we have released a new API called the “<strong>Live Stream Detailed Call History</strong>”, which is accessible from the <a>/v1/cdr_stream</a> endpoint. When you view the documentation for this API <a>here</a>, it states that Cisco recommends using this new endpoint “as the primary API for customers who need to continuously consume CDRs for custom reporting, analytics applications, or integration into internal data stores.” But what exactly is the difference between the two?</p>
<h4><a class="anchor-tag icon icon-link_16 icon-link" name="lets-take-a-look-at-the-documentation-title"></a>Let’s take a look at the documentation…</h4><a class="offset-anchor-tag" name="lets-take-a-look-at-the-documentation"></a><blockquote>
<p>The <code>CDR Stream</code> endpoint will allow customers to call the API as soon as the call ends, typically 1 minute before the current time. It automatically backfills any late records as soon as they are processed in the Webex Calling cloud. These records are then delivered in subsequent API calls, ensuring reliable CDR delivery.</p>
<p>To retrieve <strong>Detailed Call History</strong> information, you must use a token with the <code>spark-admin:calling_cdr_read</code> scope. The authenticating user must have the administrator role &quot;<strong>Webex Calling Detailed Call History API access</strong>&quot; enabled.</p>
<p><strong>Detailed Call History</strong> information is available 1 minute after the call ends and the data reaches Webex Calling cloud. Records may be retrieved for up to 12 hours from the time they are stored. For example: if a call's data reaches the Webex Calling cloud at 9:46 AM, the record becomes available from 9:47 AM and remains retrievable until 9:46 PM the same day. Additionally, only 2 hours’ worth of records can be retrieved per request (i.e., 2 hours between the selected start and end times in a single API call).</p>
</blockquote>
<p>For those who have used the <code>/v1/cdr_feed</code> endpoint, that sounds very similar to the way that endpoint worked, although the 12-hour retention and 2-hour window is new. Viewing the documentation for that endpoint, we see:</p>
<blockquote>
<p>The <code>CDR Feed</code> API can query any 12-hour period between 5 minutes ago and 30 days prior to the current UTC time. Only 12 hours of records can be retrieved per request (i.e., the time between the selected start and end times in a single API call). For example: If a call ends at 9:46 AM, the record is available for collection starting at 9:51 AM and remains available until 9:46 AM 30 days later. The maximum query duration starting at 9:51 AM would end at 9:51 PM the same day.</p>
</blockquote>
<p>At first glance, that seems like the better endpoint to use. You can pull longer windows of time (12 hours vs. 2 hours) and can get data from as far back as 30 days. So why would Cisco recommend the &quot;<strong>CDR Stream</strong>&quot; instead? Both endpoints need a <code>startTime</code> and <code>endTime</code> parameter. Both support a <code>locations</code> parameter to limit the records to specific Locations, and both allow you to specify a <code>max</code> number of records per page between 500 and 5000 records. So, what is the difference?</p>
<p>To answer that, you need to understand the timestamps of the records themselves. A CDR contains a field called &quot;<strong>Start Time</strong>&quot;, which indicates when the call represented by the CDR began. There is no field in the CDR itself that indicates when the record was written to the database, so even though you may have a record with a <strong>Start Time</strong> of 11:59, it may not have been written to the CDR database until 12:00, or later if there were any delays in processing the record. When using the <code>/v1/cdr_feed</code> endpoint, the <code>startTime</code> and <code>endTime</code> parameters refer specifically to this &quot;<strong>Start Time</strong>&quot; field within the CDR. The <code>/v1/cdr_stream</code> endpoint, on the other hand, uses the <code>startTime</code> and <code>endTime</code> parameter to refer to <strong>the exact time the record was written to the database</strong>.</p>
<h6>Which CDR API should I use?</h6><p>Let's think of a real-world example you might encounter. Imagine a script that runs every hour, right at the top of the hour and pulls the CDRs from the previous hour. If a call is placed at 11:59 but lasts 10 minutes, the CDR will not be written to the database until 12:10, one minute after the call ends. The script would have run at 12:00 to retrieve records from 11:00 to 12:00. As you would expect, but probably hadn't thought about before now, that call would not be returned in the response since the record isn't written until ~1 minute after the call ends. But, since the <code>startTime</code> and <code>endTime</code> parameters refer to the &quot;<strong>Start Time</strong>&quot; field of the CDR itself (which would be 11:59 in this case, because that's when the call started), the script execution at 13:00, which would request calls from 12:00-13:00, would also not return that call. In order to find it using the <code>cdr_feed</code> endpoint, you would have to overlap the time windows you are searching for, which could lead to a lot of duplicate entries and wasted processing time, in addition to still leaving a potential gap if a call lasted longer than your query window.</p>
<p>Knowing that the new <code>cdr_stream</code> endpoint uses the timestamp when the record was written to the database, it should be clear that this approach is much more reliable. In the previous example, the record would be written to the database at 12:10, so the record will appear in the 12:00-13:00 records, even though the call started during the 11:00-12:00 hour. And even if the record would have been delayed in processing, the record will always be contained in the &quot;new&quot; records the script is requesting, ensuring no records are lost. The documentation refers to automatically backfilling records and this is what it is referring to. You no longer need to manually backfill missing records with subsequent calls to <code>cdr_feed</code> with the same time window you used previously.</p>
<p>Now that we understand why this new endpoint is more reliable and why Cisco recommends it to &quot;continuously consume CDRs for custom reporting, analytics applications, or integration into internal data stores&quot;, you may be wondering why Cisco didn't simply replace the existing <code>cdr_feed</code> logic with this new logic. Why is the old <code>cdr_feed</code> still valuable?</p>
<p>To see the value of having both endpoints, let's go back to the example call we have been discussing. If a developer receives a request from their business to create a report that shows &quot;all calls placed or received between 11:00 and 12:00&quot;, and that request comes days after the calls have occurred, let's think about what would be returned by each endpoint. As you've seen, using <code>cdr_stream</code> with a <code>startTime=11:00</code> and <code>endTime=12:00</code> would not contain the call from our examples. Instead, using the <code>cdr_feed</code> endpoint with the same <code>startTime</code> and <code>endTime</code> would ensure that all calls placed or received during that window will be returned.</p>
<p>Each endpoint has its own benefit, but the two will likely not be used by the same process. Automated systems that use the <code>cdr_stream</code> endpoint will likely insert that data into a data store where the data can be queried directly, eliminating the need for the <code>cdr_feed</code> endpoint. Developers who wish to collect reporting data from Webex would benefit from <code>cdr_feed</code>.</p>
<h6>Conclusion</h6><p>As you can see, understanding the differences between these endpoints is essential when deciding which to use for your development needs, but the following advice serves as a good rule of thumb:</p>
<ul>
<li>For continuous processes to pull CDRs from Webex into another data store, use the <code>cdr_stream</code> endpoint</li>
<li>For historical reporting within 30 days, use the <code>cdr_feed</code> endpoint</li>
<li>For historical reporting longer than 30 days, or for larger windows of time, the existing <code>/v1/reports</code> endpoint should be used. This endpoint has not changed.</li>
</ul>
